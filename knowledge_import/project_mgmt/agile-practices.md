# 敏捷开发实践

## Scrum框架

### Sprint规划

**步骤**：
1. **待办事项梳理**：团队共同审查产品待办列表
2. **任务估算**：使用故事点估算工作量
3. **选择任务**：根据团队容量选择本轮任务
4. **任务分解**：将大任务拆分为可执行的小任务

**估算方法**：
- **计划扑克**：团队成员匿名投票估算
- **相对估算**：与已知任务比较
- **T-shirt sizing**：XS/S/M/L/XL粗略估算

### 每日站会

**时间**：15分钟以内

**三个问题**：
1. 昨天完成了什么？
2. 今天要做什么？
3. 遇到了什么障碍？

**原则**：
- 站着开会，保持简短
- 只同步状态，不解决问题
- 问题会后单独讨论

### Sprint评审

**时间**：1-2小时

**内容**：
- 演示完成的功能
- 收集用户反馈
- 讨论改进方向

**参与人员**：
- 产品负责人
- 开发团队
- 利益相关者

### Sprint回顾

**时间**：30分钟-1小时

**目的**：
- 回顾本轮工作
- 识别改进点
- 制定改进计划

**常见问题**：
- 哪些做得好？
- 哪些需要改进？
- 下轮如何改进？

## Kanban方法

### 看板设计

**列定义**：
- **待办**：等待处理的任务
- **进行中**：正在处理的任务
- **代码审查**：等待审查的任务
- **测试**：正在测试的任务
- **完成**：已完成的任务

**WIP限制**：
- 限制每个列的最大任务数
- 防止过度并行
- 提高流动效率

### 工作流程

**核心原则**：
1. **可视化**：所有任务一目了然
2. **限制WIP**：避免过度负载
3. **持续流动**：保持任务快速流转
4. **度量改进**：持续优化流程

**流程优化**：
- 识别瓶颈
- 减少阻塞
- 优化协作

## 用户故事

### INVEST原则

- **I**ndependent：独立可交付
- **N**egotiable：可协商
- **V**aluable：有价值
- **E**stimable：可估算
- **S**mall：足够小
- **T**estable：可测试

### 故事编写模板

```
作为[角色]，
我希望[功能]，
以便[价值]。
```

**示例**：
```
作为用户，
我希望能够重置密码，
以便忘记密码时可以重新访问账户。
```

### 故事拆分

**拆分策略**：
1. **按功能拆分**：按功能模块拆分
2. **按流程拆分**：按业务流程拆分
3. **按复杂度拆分**：按技术复杂度拆分
4. **按优先级拆分**：按优先级顺序拆分

**拆分技巧**：
- 保持拆分后故事的独立性
- 确保每个故事都有价值
- 控制故事大小在1-3个故事点

## 估算技术

### 故事点

**定义**：相对工作量的度量单位

**常用尺度**：
- Fibonacci序列：1, 2, 3, 5, 8, 13
- 幂次序列：1, 2, 4, 8, 16

**估算规则**：
- 团队共同估算
- 基于历史数据校准
- 避免精确估算

### 容量规划

**计算团队容量**：
```
团队容量 = 团队人数 × 每人可用天数 × 效率系数
```

**示例**：
```
团队人数：5人
每人可用天数：5天/周
效率系数：0.8

团队容量 = 5 × 5 × 0.8 = 20个故事点
```

## 敏捷度量

### 速度

**定义**：团队在一个Sprint内完成的故事点数

**用途**：
- 预测未来Sprint的产出
- 评估团队效率
- 识别改进机会

**注意**：
- 速度不是绩效指标
- 不要比较不同团队的速度
- 关注趋势而非单点数值

### 燃尽图

**类型**：
- **产品燃尽图**：显示产品待办列表的剩余工作量
- **Sprint燃尽图**：显示Sprint待办列表的剩余工作量

**解读**：
- 理想线：线性下降
- 实际线：根据实际进度调整
- 偏差分析：识别进度问题

### 累积流量图

**用途**：
- 可视化工作流程
- 识别瓶颈
- 预测交付时间

**指标**：
- **周期时间**：任务从开始到完成的时间
- **吞吐量**：单位时间完成的任务数
- **WIP**：在制品数量

## 敏捷团队

### 角色定义

**产品负责人**：
- 定义产品愿景
- 管理产品待办列表
- 代表用户利益

**Scrum Master**：
- 引导Scrum流程
- 移除团队障碍
- 促进团队协作

**开发团队**：
- 自组织完成任务
- 跨职能协作
- 持续改进

### 团队协作

**沟通方式**：
- 面对面沟通
- 每日站会
- 即时通讯工具

**协作工具**：
- Jira：任务管理

- Slack：团队沟通

## 常见问题

### 需求变更

**处理策略**：
- 将变更添加到产品待办列表
- 在Sprint评审时讨论
- 根据优先级安排到后续Sprint

### 任务延期

**处理策略**：
- 及时沟通风险
- 重新评估任务
- 调整计划

### 团队冲突

**处理策略**：
- 及时发现冲突
- 促进开放沟通
- 寻求第三方调解

## 敏捷转型

### 转型步骤

1. **培训**：团队学习敏捷知识
2. **试点**：选择一个团队试点
3. **调整**：根据反馈调整流程
4. **扩展**：推广到更多团队

### 成功因素

- **高层支持**：获得管理层支持
- **团队意愿**：团队积极参与
- **持续改进**：不断优化流程

## 总结

敏捷开发是一种灵活的项目管理方法，通过迭代和增量交付，快速响应用户需求变化。关键在于团队协作、持续改进和拥抱变化。
