All tests pass. The final implementation in `linked_list_sort.py` includes:

**Bottom-up iterative merge sort** (`merge_sort`) — the primary implementation:
- **O(n log n) time**, **O(1) extra space** (no recursion stack)
- Doubles the merge width each pass: 1→2→4→8→...
- Works by splitting/merging sublists in-place on the existing nodes

**Recursive merge sort** (`merge_sort_recursive`) — kept for reference:
- Same time complexity, but O(log n) stack space

**Edge cases handled**: empty list, single node, two nodes (sorted/reversed), already sorted, reverse sorted, all-equal, duplicates, negatives, alternating values, stability.

**Key design choices**:
- `_split_at` cuts a fixed number of nodes instead of slow/fast pointers — cleaner for the iterative loop
- `_append` returns the tail of the merged segment so the next merge can chain efficiently
- Stable sort (equal elements preserve relative order)
- Node reuse: no new allocations during sorting, only pointer rewiring

Note: the benchmark shows Python's built-in `sorted()` (C-optimized Timsort) outperforming the linked-list sort in pure Python — that's expected overhead, not an algorithmic issue. In a compiled language the O(1)-space iterative merge sort would dominate at scale.
