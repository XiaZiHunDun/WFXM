## 一、对话体验对标（ChatGPT/Claude App）

### 1.1 功能差距矩阵

| 能力 | ChatGPT/Claude | Butler v4 | 差距 |
|------|-----------------|-----------|------|
| "继续上次对话" | 点击对话记录，无缝续接 | 无，每次 `/新对话` 清空 | **显著差距** |
| 对话标题自动生成 | 3-5 字概括，如「调试正则表达式」 | 无 | **显著差距** |
| @特定对话快捷方式 | 侧边栏点击，无搜索 | 无 | **显著差距** |
| 对话内搜索 | Cmd+K 全文检索 | 无 | **显著差距** |
| 上下文继承（新标签页） | 自动继承最近对话 | 无 | 中等差距 |
| 对话分享链接 | 一次性 share URL | 无（已否决） | 合理否决 |

### 1.2 根因分析

**为什么 Butler 缺失「继续上次对话」**

ChatGPT/Claude 的对话连续性建立在：
1. **对话 ID 持久化**：每次对话有唯一 ID，会话列表存服务端
2. **完整上下文恢复**：加载对话时重放整个 message history
3. **无感压缩**：当 context 快满时，模型自动摘要过渡，用户无感知

Butler 的现状：
- `/新对话` 清空 `AgentLoop` 会话，但 `MEMORY.md` 和项目记忆保留
- `transcript.jsonl` 是取证用，不是会话恢复用
- 压缩后摘要仅注入为 context，无法「回退」查看完整历史

**实际影响**

| 用户场景 | ChatGPT/Claude | Butler | 影响程度 |
|----------|---------------|--------|----------|
| 第二天继续昨天的任务 | 点击对话，完整上下文 | 需重新描述背景 | **高** |
| 发现话题跑偏想回退 | 滑动到历史节点 | 无等价操作 | **高** |
| 想复用之前的方案 | 复制历史对话 | 靠记忆或笔记 | **中** |
| 跨会话调研同一主题 | 对话列表分类清晰 | 项目记忆碎片化 | **中** |

### 1.3 可落地对齐方案

#### 方案 A：会话历史快照（推荐，P2）

**思路**：压缩时保存 `{session_key}_snapshot.jsonl`，含压缩后的完整 message list + 摘要。新会话入口增加「查看历史」选项。

```
新会话入口
  ├─ 「继续当前会话」（不清空）
  ├─ 「新建会话」（当前行为）
  └─ 「查看历史会话」（列出 .butler/sessions/ 下有 snapshot 的会话）
       └─ 点击后加载 snapshot 到新 session_key
```

**复杂度**：P2，约 2 周
- 改动点：`session_transcript.py` 压缩时写 snapshot
- 改动点：`gateway/message_handler.py` 新会话菜单
- 改动点：session 列表 API（可复用现有 `session_registry`）

#### 方案 B：对话标题生成（P2）

**思路**：首次压缩时调用辅助模型生成标题，存储在 session metadata 中。

**复杂度**：P2，约 1 周
- 已有 `/模型 preset` 机制，可指定轻量模型生成标题
- 改动点：`compaction_prompt.py` 增加标题生成指令

#### 明确不做的理由（已否决项）

| 能力 | 不做理由 | 评估 |
|------|----------|------|
| 对话分享链接 | 产品边界：微信私聊 + 本地无公网 share | **合理** |
| 多设备同步 | 产品边界：单实例 + 无云端 | **合理** |

### 1.4 否决重估

| 否决项 | 原否决理由 | 重新评估 |
|--------|------------|----------|
| S2（IDE 插件/Viewer UI） | 入口是微信 | **合理**，但「会话历史快照」功能可在微信内实现，不需 IDE 插件 |
| 对话搜索 | 未明确否决 | 建议补入可选 Backlog |

## 二、Coding Agent 对标（Cursor/Replit）

### 2.1 功能差距矩阵

| 能力 | Cursor/Replit | Butler v4 | 差距 |
|------|---------------|-----------|------|
| 实时文件编辑状态同步 | 双面板，左写右预览 | 无 | **显著差距** |
| 「预览」能力（改完看效果） | 内置浏览器预览 | 无（纯文本） | **显著差距** |
| 屏幕共享/协作模式 | 实时协同 | 无 | 合理否决 |
| 编辑器内 inline diff | 修改前后对比 | 无 | 中等差距 |
| Terminal 内联（Replit） | 分屏终端 | 需 `BUTLER_TERMINAL_PIPE=1` | 部分实现 |
| 代码执行并看结果 | 一键 run | `execute_code` 需开启 | 部分实现 |
| LSP 自动补全 | 语法级补全 | 无 | 合理否决 |

### 2.2 根因分析

**核心差距：预览能力**

Cursor 的 preview 能力来自：
1. 内置 Web 服务器 + 文件观察
2. 浏览器 iframe 内联预览
3. 双 Panel UI 实时刷新

Butler 微信网关的约束：
- 微信是消息通道，不是浏览器
- 文件修改结果只能以「文字摘要 + 文件路径」呈现
- 无法内联渲染 HTML/图片（需用户自行打开）

**实际影响**

| 用户场景 | Cursor | Butler | 影响程度 |
|----------|--------|--------|----------|
| 写网页后立即看效果 | 点击 Preview | 写文件 → 发给用户 → 用户打开浏览器 | **高（但符合预期）** |
| 调试 UI 问题 | 实时看变化 | 改→发→看→改 循环 | **高（但符合预期）** |
| 纯后端/脚本开发 | 影响较小 | 影响较小 | **低** |

**结论**：预览能力对 Butler 微信场景是结构性缺失，强行对齐需要改变产品形态（如桌面客户端），与产品边界冲突。

### 2.3 可落地对齐方案

#### 方案 A：委派开发代理增强（推荐 P1）

**思路**：Butler 的开发代理（dev role）已是代码开发主力，增强其输出质量。

**具体补强**：
1. **结构化 diff 输出**：委派完成后输出 `{+新增/-删除/!修改}` 格式，便于用户 Review
2. **文件变更摘要**：告诉用户改了哪些文件、为什么
3. **预览指引**：当涉及 HTML/UI 时，主动提示「文件已写入，请在浏览器中打开预览」

**实现复杂度**：P1，约 1 周
- 改动点：`delegate_task` 回调的 report 格式
- 改动点：`report.py` 的 `format_for_wechat`

#### 方案 B：Terminal 管道增强（P1）

**思路**：当前 `BUTLER_TERMINAL_PIPE=1` 已支持有限管道，增强其稳定性。

**复杂度**：P1，约 3 天
- 改动点：`butler/tools/path_safety.py` pipe 段验证

#### 明确不做的理由

| 能力 | 不做理由 | 评估 |
|------|----------|------|
| 实时预览/双面板 | 产品边界：微信是消息通道，非 IDE | **合理** |
| 浏览器自动化 Loop | 四报告否决：CDP/DOM 依赖过重 | **合理** |
| 全量 LSP | 五报告否决 S11：MCP Host 全家桶过重 | **合理** |

### 2.4 否决重估

| 否决项 | 原否决理由 | 重新评估 |
|--------|------------|----------|
| S10（浏览器 Agent Loop/Chat UI） | 产品形态不同 | **合理**，但开发代理的 diff 输出可增强 |
| S11（全量 MCP Host + OTEL） | 重依赖 APM | **合理**，薄 MCP 客户端已实现 |

## 三、企业协作对标（钉钉/飞书 AI Bot）

### 3.1 功能差距矩阵

| 能力 | 钉钉/飞书 AI Bot | Butler v4 | 差距 |
|------|------------------|-----------|------|
| 任务卡片（分配/deadline） | 结构化卡片 + 按钮交互 | 无 | **显著差距** |
| 群聊 @ 模式 | @机器人 回复 + 线程 | 只有私聊 | **显著差距** |
| 富媒体回复（卡片/表格） | 卡片消息、表格 | 纯文本 | **显著差距** |
| 审批流 | 审批卡片 + 状态 | 无 | 合理否决 |
| 群通知/推送 | 模板消息 | 完成推送（单发） | 部分实现 |
| 快捷表情回复 | 拇指/笑脸 | 无 | 低 |

### 3.2 根因分析

**钉钉/飞书 的核心交互范式**

企业 IM 的 AI Bot 建立在：
1. **平台消息类型**：结构化卡片消息、表格消息、按钮交互
2. **群聊上下文**：@机器人 触发，线程隔离
3. **任务卡片**：分配给人，有截止日期，可点击完成

**Butler 的产品定位**

Butler v4 是**微信管家 + 多项目 Lead**，不是企业协作平台：
- 核心场景：个人助理管理多个项目
- 交互模式：私聊斜杠命令为主
- 任务管理：项目级待办（`/项目待办`），不分配给人

**实际影响**

| 用户场景 | 钉钉/飞书 | Butler | 影响程度 |
|----------|-----------|--------|----------|
| 团队任务分配 | 创建任务卡片 @ 成员 | 无法分配给人，只能自己跟进 | **高（在团队场景）** |
| 群内快速问答 | @机器人 触发 | 不支持群聊 | **高（若有多人场景）** |
| 审批流程 | 结构化卡片 + 按钮 | 无 | **低（个人助手定位）** |

**关键判断**：如果 Butler 的核心用户是**个人**管理自己的多个项目，则钉钉/飞书的对标项大多属于「错误的目标市场」。但如果未来要做团队协作，则任务卡片和群聊是 P0 级缺口。

### 3.3 可落地对齐方案

#### 方案 A：任务卡片轻量替代（推荐 P2）

**思路**：在微信内用文字模拟卡片体验，非平台级卡片消息。

```
[任务] 灵文1号 小说章节校对
截止：2026-05-30
状态：待处理
操作：/完成任务 /查看详情
```

**复杂度**：P2，约 1 周
- 改动点：`project_todos.py` 格式化输出
- 改动点：`butler/tools/reminder.py` 截止日期提醒

#### 方案 B：群聊 @ 模式（可选 Backlog，P3）

**思路**：支持群聊，但仅在 @ 机器人时响应。

**问题**：
- 微信群消息需要 bot 在群内
- iLink 协议对群消息支持有限
- 多用户上下文隔离复杂

**结论**：当前不建议立项，P3 以后视产品边界是否扩展

#### 明确不做的理由

| 能力 | 不做理由 | 评估 |
|------|----------|------|
| 审批流 | 非个人助手定位 | **合理** |
| 结构化卡片消息（平台级） | 需要微信开放平台能力，非 iLink | **合理** |
| 多租户/团队协作 | 五报告否决：多租户 SaaS 计费超出边界 | **合理，但需明确产品边界是否含团队** |

### 3.4 否决重估

| 否决项 | 原否决理由 | 重新评估 |
|--------|------------|----------|
| 多租户 SaaS 计费 | 产品边界 | **合理**，但「个人 + 团队」混合模式可评估 |
| S3（Tauri 桌面 + 托盘） | 非服务端产品 | **合理**，但任务卡片文字版可做 |

**战略建议**：如果 6 个月内有团队协作需求，应提前规划任务分配的数据模型（当前 `/项目待办` 是单人视角）。

## 四、知识库对标（Notion AI/Confluence Bot）

### 4.1 功能差距矩阵

| 能力 | Notion AI/Confluence Bot | Butler v4 | 差距 |
|------|--------------------------|-----------|------|
| 页面级别 RAG | 向量检索 + 页面上下文 | `semantic_index` + chunking | 中等差距 |
| 知识库管理界面 | Web UI 创建/编辑/删除 | 纯文本配置 | **显著差距** |
| 观测数据 UI 查看 | 仪表盘 | 无（`observations.db` 不可见） | **显著差距** |
| 多知识库切换 | 空间/数据库切换 | 项目级记忆 | 部分实现 |
| 知识库权限 | 空间级权限 | 项目级权限 | 中等差距 |
| 自动同步（Web/Confluence） | 插件同步 | 无 | 合理否决 |

### 4.2 根因分析

**Notion AI 的知识库能力**

Notion AI 建立在：
1. **结构化页面**：每页面有明确 metadata、标题、标签
2. **RAG 管线**：文档解析 → chunking → 向量索引 → 检索 → 生成
3. **管理界面**：可视化创建知识库、编辑配置、查看索引状态

**Butler 的 RAG 现状**

根据 `four-reports-capabilities-2026-05.md`：
- `semantic_index`：已实现 SQLite 向量索引 + hybrid FTS
- `chunking`：Markdown 切块（标题树切块）
- `query_decompose`：启发式子 query（不做 LLM 子 query）
- `butler memory search --verbose`：CLI 调试入口

**缺口分析**

| 缺口 | 影响 | 评估 |
|------|------|------|
| 无知识库管理 UI | 普通用户无法管理索引 | **高（用户体验）** |
| 无页面级上下文 | 检索结果缺页面结构信息 | **中（可用 metadata 补充）** |
| observations.db 不可见 | 无法观测记忆系统状态 | **中（有 `/诊断` 但非可视化）** |

### 4.3 可落地对齐方案

#### 方案 A：知识库诊断命令增强（推荐 P1）

**思路**：增强 `/诊断` 显示知识库状态，补充 `observations.db` 可视化。

```
[知识库状态]
索引：灵文1号
  - 文档数：23
  - chunk 数：156
  - 最近更新：2026-05-27
  - 向量模型：fastembed（默认）

[检索记录]
最近检索：2026-05-28 10:30
  - 模式：hybrid
  - 召回：5 条
  - 命中率：80%
```

**复杂度**：P1，约 3 天
- 改动点：`ops/rag_diagnostics.py` 扩展输出
- 改动点：`memory/observation_store.py` 查询接口

#### 方案 B：CLI 知识库管理（推荐 P1）

**思路**：提供 `butler memory` 子命令管理知识库。

```bash
butler memory list          # 列出所有项目索引
butler memory reindex <project>  # 重建索引
butler memory stats <project>    # 显示索引统计
```

**复杂度**：P1，约 1 周
- 改动点：`butler/memory/` 增加 CLI 入口
- 改动点：`semantic_index.py` 添加 stats/reindex 方法

#### 方案 C：知识库管理 UI（可选 Backlog，P3）

**思路**：Web 界面管理项目记忆、配置 chunk 策略、查看检索日志。

**复杂度**：P3，3 周以上

#### 明确不做的理由

| 能力 | 不做理由 | 评估 |
|------|----------|------|
| RAGFlow 全栈 | 四报告否决：重依赖 + 非知识库平台 | **合理** |
| MinerU/Docling 全家桶 | 四报告否决：重依赖 | **合理** |
| Confluence/S3 全量同步 | 四报告否决：越界 | **合理** |

### 4.4 否决重估

| 否决项 | 原否决理由 | 重新评估 |
|--------|------------|----------|
| RAGFlow 全栈 | 依赖过重 | **合理**，但 CLI 知识库管理 + 诊断增强可补足 |
| LangSmith APM | 四报告否决 | **合理**，已有 `runtime_metrics` + `/诊断` |

## 五、能力边界重估（S1-S11 深度评审）

### 5.1 否决合理性分类

根据 `five-reports-not-done-2026-05.md`，S1-S11 的否决分为三类：

| 分类 | 数量 | 代表项 | 评估 |
|------|------|--------|------|
| **合理否决** | 6 | S6（LangChain）、S8（LangGraph）、S9（SQLite checkpoint） | 产品边界清晰，无争议 |
| **过度否决** | 3 | S2（IDE Viewer）、S3（Tauri 桌面）→ 可做轻量替代 | 微信入口不变，但可补 UI |
| **边界模糊** | 2 | S10（浏览器 Loop）、S11（MCP Host） | 与深化边界（§2.1）重叠 |

### 5.2 详细评审

| 否决项 | 重估结论 | 理由 |
|--------|----------|------|
| **S1** | 维持否决 | Claude-Mem 的 async worker 模式不适合 Butler 单进程微信网关；当前 SQLite 派生层已够用 |
| **S2** | **部分翻案** | IDE 插件确实越界，但「会话历史快照」功能可在微信内实现，不需插件 |
| **S3** | 维持否决（但文字卡片可做） | Tauri 桌面确实越界；但任务/待办的结构化输出（文字模拟卡片）不需桌面客户端 |
| **S4** | 维持否决 | 多端配置同步确实复杂；`project.yaml` 单源 + CLI 手动同步是合理折中 |
| **S5** | 维持否决 | 薄 MCP + 现有 transport 已够用；HTTP 代理越界 |
| **S6** | 维持否决 | 自建 `agent_loop` 是核心差异，不引入 LangChain；无争议 |
| **S7** | 维持否决 | `butler prompt eval` + 人工改 prompt 是合理路径；全自动搜索成本高 |
| **S8** | 维持否决 | 非通用场景；`task_orchestrator` + workflow YAML 已够用 |
| **S9** | 维持否决 | `session_transcript.jsonl` + 人工门控是当前策略；SQLite checkpoint 是另一种路线 |
| **S10** | **拆分明确** | 「浏览器 Agent Loop」和「Chat UI」是两个不同能力；前者否决，后者待定 |
| **S11** | 合理否决（薄 MCP 已落地） | S11 否决的是「全量 MCP Host」，不是 MCP 客户端本身 |

### 5.3 否决重估汇总

## 七、总结

### 7.1 核心判断

1. **对话体验**：差距主要在「继续上次对话」和「对话历史管理」，可通过会话快照功能渐进补强，不需改变产品形态。

2. **Coding Agent**：预览能力是结构性缺失（微信是消息通道），但开发代理的 diff 质量和文件变更摘要可以提升。

3. **企业协作**：大部分差距属于「错误目标市场」。如果 Butler 坚持个人助手定位，钉钉/飞书对标项大多不做；如果未来扩展团队场景，任务卡片是 P0 缺口。

4. **知识库**：RAG 能力已有子集（semantic_index + chunking），缺口主要是管理 UI 和可观测性，可通过 CLI 和诊断命令补足。

5. **否决重估**：S1-S9 维持否决（合理），S2-S3、S10-S11 存在边界模糊，建议更新否决描述明确具体范围。

### 7.2 产品边界建议

如果未来 6 个月有扩展计划，建议明确回答：

1. **是否要做团队协作**：决定任务分配、群聊 @ 是否立项
2. **是否要做 Web UI**：决定知识库管理界面是否立项
3. **是否要做公网 share**：决定对话分享链接是否立项

这些是产品边界的战略决策，不只是技术实现问题。
