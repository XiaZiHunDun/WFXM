# 产品挑刺报告

> 日期：2026-05-28
> 范围：Butler 系统全方位产品审视
> 维度：用户体验、功能完整性、竞品对标、信任可靠性、信息架构、商业场景

---

## 一、用户体验（UX）

### 1. 入口认知门槛过高
- **问题**：产品定位"多项目 AI 协助系统"对普通用户而言太技术化
- **表现**：微信作为核心入口，但用户心智中微信=聊天工具，Bot交互需要"发现"这个管家存在；`butler chat` CLI 无引导性 welcome flow，直接进入黑屏等待
- **影响**：新用户首次使用流失率高

### 2. 命令体系碎片化
- **问题**：`/新对话`、`/详细`、`/诊断`、`/workflow`、`/budget` 等命令无统一帮助入口
- **表现**：用户不知道"能做什么"，只能靠记忆或踩坑
- **建议**：统一的 `/help` 或一级菜单展示可用能力

### 3. 错误消息过于技术化
- **问题**：LLM 调用失败时往往暴露内部堆栈，对用户无意义
- **表现**："context_pipeline hygiene preflight failed"、"llm_retry schema recovery" 等代码级消息
- **建议**：微信端统一翻译为用户可操作的指引

### 4. 反馈延迟无预期管理
- **问题**：微信消息发送后无"正在思考"状态
- **表现**：复杂任务可能需要 30s+，用户不确定 Bot 是否收到；工具调用过程对用户是黑箱
- **建议**：增加状态提示（如"正在检索记忆库..."）

---

## 二、功能完整性

### 5. 项目发现与切换体验差
- **问题**：`projects/` 目录下有多个项目，但微信端无法直观浏览
- **表现**：切换项目靠 `/切 <项目名>` 精确匹配，输错就找不到；无项目列表命令
- **建议**：增加 `/项目列表` 命令

### 6. 委派结果摘要质量不稳定
- **问题**：`/详细` 输出的摘要依赖 LLM 压缩，关键信息可能丢失
- **表现**：长时间对话后委派，早期上下文已被压缩，摘要可能包含"根据之前的讨论"但用户已不记得
- **建议**：摘要增加时间戳和上下文版本标识

### 7. 记忆系统断点
- **问题**：`/新对话` 清空会话，但 `.butler/observations.db` 和 workspace 记忆是否保留？用户无感知
- **表现**：新对话后立刻问"上次那个bug修了吗"，Bot 可能因记忆不一致而答非所问；跨会话任务连续性无保障
- **建议**：新对话入口增加记忆状态说明

### 8. Workflow 调试成本高
- **问题**：Workflow YAML 定义后，无法在微信端预览 DAG 结构
- **表现**：执行失败后用户不知道哪一步出问题；内置 workflow 无文档说明入口
- **建议**：增加 `/workflow <名称> 预览` 命令

| "继续上次对话" | 无，每次独立会话 |
| 对话标题自动生成 | 无 |
| @特定对话快捷方式 | 无 |

### 10. 相比 Cursor/Replit Agent
| 竞品能力 | Butler 现状 |
|----------|-------------|
| 实时文件编辑状态同步 | 无 |
| "预览"能力（改完看效果） | 无 |
| 屏幕共享/协作模式 | 无 |

### 11. 相比钉钉/飞书 AI Bot
| 竞品能力 | Butler 现状 |
|----------|-------------|
| 任务卡片（分配/deadline） | 无 |

## 四、信任与可靠性

### 13. 敏感操作无二次确认
- **问题**：`patch_file`、`write_file`、`delete_file` 等写操作直接执行
- **表现**：危险命令经 guardrails 但用户无感知
- **建议**：微信端写操作增加"请回复 Y 确认"机制

### 14. 能力边界用户不清晰
- **问题**：v4 架构文档有大量"否决"条目，但用户不知道这些限制
- **表现**：哪些能做哪些不能做，需要深入阅读文档或实际踩坑
- **建议**：增加 `/能力` 命令，展示白名单

### 15. 日志黑箱
- **问题**：执行出错后，`/诊断` 只给指标，无实际排查路径
- **表现**：`logs/` 目录无 WebUI，普通用户无法自主排查
- **建议**：`/诊断` 增加错误类型分级和基本排查步骤

---

## 五、信息架构

### 16. 文档与产品脱节
- **问题**：大量架构文档（`v4-architecture.md`）但普通用户看不到
- **表现**：`AGENTS.md` 内容未转化给用户侧；技术术语未抽象为用户语言
- **建议**：用户侧帮助文档与产品同步更新

### 17. 多入口体验不一致
- **问题**：CLI 和微信命令集合略有差异
- **表现**：CLI 有流式输出，微信只有最终结果；状态（项目、记忆）在不同入口间不共享
- **建议**：明确各入口的功能 parity 和差异文档

---

## 六、商业与场景

### 18. 单用户设计，无法团队协作
- **问题**：无多用户权限体系
- **表现**：无法分配任务给其他成员；无审计日志查看谁做了什么操作
- **建议**：规划多租户架构（已在 roadmap 否决，需重新评估）

### 19. 未覆盖移动端核心场景
- **问题**：推送消息无法主动触达用户
- **表现**：长时间运行任务完成时用户可能不在微信前；无移动端通知扩展
- **建议**：可选：微信模板消息/服务号通知
