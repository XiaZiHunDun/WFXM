### 3. 中部：真实 Bug（24pt 标题 + 18pt 正文）

```
🐛 真实 Bug · commit 323862e

现象  ：transcript_fts.py 的 PRIMARY KEY 多了一个右括号
根因  ：SQLite 早期版本静默吞语法错
后果  ：FTS 索引重建失败，但代码以为成功 → 所有 INSERT 静默失败
修法  ：去掉多余 `)` + 加显式 health check
教训  ：第三方容错是双刃剑
```

| 多项目切换  | 无         | N 倍（独立 MEMORY）|
| 微信接入   | 无         | ∞（原生网关）|

→ "直接 LLM 适合聊天，不适合远程多项目管理"

## 五、可能追问预案

| 追问 | 锚点 |
|------|------|
| Recall@3 ≥ 0.8 怎么测的？ | "`embedding_health.py` 实测，候选集 = MEMORY + 项目档案，q=test set of 50 真实查询" |
| 1254 是全量吗？ | "`find butler -name '*.py' \| wc -l`，含 `__init__.py` 但不含 tests/" |
| 232 / 176 怎么算的？ | "`find butler/core -name '*.py' \| wc -l` = 232；gateway 同理" |
| 测试覆盖 ~80% 准吗？ | "`pytest --cov=butler` 实测，已写测试 + 集成测试覆盖主路径" |
| 还有别的 bug 吗？ | "还有 croniter 缺失被 try/except 吞掉、压缩丢锚点（修法 = 压前抽 fact）" |
| 为什么不修 SQLite 版本？ | "兼容老版本是策略——加 health check 比升 SQLite 版本成本低、风险小" |
