# 05 · WFXM Butler · 一句话技术标签

> **用途**：PPT 封面、面试开场、结尾收口——这一句话是整个项目的"灵魂锚点"。

---

## 0. 一句话技术标签（已确认）

> **"跨进程工具路由 + 多源记忆治理"**

### 这句话的拆解

| 词 | 含义 | WFXM 对应 |
|----|------|-----------|
| **跨进程** | 工具调用跨 owner / 子代理 / runtime / 外部 MCP 多个进程边界 | `delegate_task` 跨 session、`cc_bridge` 跨 Claude Code、`mcp_self_service` 跨 MCP 进程 |
| **工具路由** | 不是"调用"，是"路由"——带权限过滤 + 上下文裁剪 + 子代理窄化 | `tools/registry.py` + `filter_tools_for_subagent` + `permissions.yaml` |
| **多源** | 记忆来自 Owner 画像 / 项目 MEMORY / 会话 transcript / 外部语料 | 三层架构 + `unified_recall` + `document_ingest` |
| **记忆治理** | 不是"存储"，是"治理"——含压缩 / fact 抽取 / 漂移检测 / 召回策略 | `fact_extraction` + `consistency_outcome` + `todos_pending_drift` + `recall_layers` |

---

## 1. 为什么是这两个词

### 为什么是"路由"而不是"调用"

- "调用"是**单点动作**（call a function）
- "路由"是**带策略的转发**（decide where to go based on context + permissions）
- 面试官听到"路由"会立刻联想到"网关 / 鉴权 / 灰度 / 限流"——这些都是 Butler 自建 Loop 的核心场景

### 为什么是"治理"而不是"管理"

- "管理"是**CRUD 操作**（save / load / delete）
- "治理"是**带规则的策略执行**（dedup / drift / fact / scope / lifecycle）
- Butler 记忆模块 90+ 文件的复杂度，本质是"治理"而非"管理"——这套话术能体现工程深度

## 2. 这句话在面试中的使用方式

### 开场（30 秒）

> "我做的项目叫 **WFXM Butler**，是一个自托管的多项目 AI 管家。如果让我用一句话记住它——
> 
> **'跨进程工具路由 + 多源记忆治理'**。
> 
> 前者解决'Agent 如何在多权限域里安全调度工具'，后者解决'Agent 如何在长会话里不丢上下文'。下面我展开讲。"

### 结尾收口（30 秒）

> "总结一下，**WFXM Butler 的核心挑战**就是这两个：
> - **跨进程工具路由**——让 100+ 工具在多 Owner、多子代理、多权限域里安全调度
> - **多源记忆治理**——让三层记忆（Owner / 项目 / 会话）在压缩、漂移、漂洗下不失真
> 
> 这两个挑战也是 Agent 行业目前最大的两个工程难题。我的方案是'自建 Loop + 文件 SSOT'，理由是......"

### Q&A 收口（如果被追问"你最大的技术贡献是什么"）

> "如果让我回答'最大的技术贡献'，我会说是**建立了'工具路由'和'记忆治理'两套工程范式**——
> - 工具路由：**2 层 15 道门**——口播精简为 5 道核心防线（permissions / 窄化 / read-before-edit / DELEGATE_BLOCKED / typed schema），但完整架构见 `02-action.md §4`
> - 记忆治理：SSOT 是文件、索引可重建、压缩前抽 fact、漂移可检测
> 
> 这两套范式不依赖特定框架，可以移植到任何 Agent 项目。"
