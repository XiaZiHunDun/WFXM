# 04 · WFXM Butler · L 栏（反思与认知升级）

> **本章定位**：拉开面试官"觉得还行"和"眼前一亮"差距的核心。**"如果重来一次，你怎么做？"是必问题。**

## 1. 架构反思（过度设计 / 设计不足）

### 1.1 过度设计（做过头的部分）

| 模块 | 当时的想法 | 现在的反思 |
|------|------------|------------|
| **`butler/memory/` 90+ 文件** | 想做"完整的认知架构"（chunking / semantic_index / observation / triplet / corrective / unified recall…） | **过度分层**。Owner 实际用到的就是三层（Owner / 项目 / 会话），其余学术化抽象增加心智负担。**重来应该砍 60%** |
| **`butler/tools/` 120+ 文件** | 工具边界按"业务域"切（delegate / memory / project / runtime / terminal / multimodal / network…） | 部分边界过早抽象。如 `delegate_*_ops.py` / `delegate_*_state.py` 拆得过细，新人找入口要先看 README |
| **三层记忆 + 多 scope 召回** | "hybrid + observation + unified recall" 三种召回 | Owner 90% 用 `--scope hybrid`。`observation` 和 `unified` 启用比例 < 5% |
| **Permission 2 层 15 道门**（入站 9 步 + 工具链 6 层） | permissions.yaml + 子代理窄化 + read-before-edit + DELEGATE_BLOCKED + typed schema + ... | **方向是对的**。但 read-before-edit 的覆盖率要靠测试守门，否则会被绕过 |

**反思一句话**：**框架代码的复杂度 ≠ 业务价值**。自建 Loop 的优势是"白盒可断点"，但每多一个抽象层就稀释一次这个优势。

### 1.2 设计不足（应该多做但没做的）

| 缺口 | 现在的痛 | 重来会怎么补 |
|------|----------|--------------|
| **可观测性** | runtime metrics 在 `/诊断` 里，但**没有 dashboard**。Owner 看健康全靠 `/诊断` + `butler doctor` 命令 | 加一个本地 web dashboard（FastAPI + 简单前端），实时看 memory hits / jobs / delegate rate |
| **跨项目搜索** | 多项目各自独立 MEMORY.md，跨项目搜必须 `butler memory search --scope hybrid`（global） | 早期就定义跨项目索引规范（虽然现在 `unified_recall` 救场） |
| **Agent 测试覆盖** | live_llm 标记除外，单测覆盖 ~80%，但**端到端 LLM 行为测试**靠 sim | 加 replay fixture + agent 输出 schema 校验（已完成 P0 #1 部分） |
| **多人协作** | 单 Owner 设计。如果团队用，需要权限/审计 | 加 Role 字段（Owner / Maintainer / Viewer） |
| **国际化** | system prompt / 工具描述全中文 | 加 i18n 抽象层（虽然没必要现在做） |

**反思一句话**：**过度设计的反面不是"少做"，而是"延后做"**。很多能力等真实使用场景出现后再加，比一开始就堆好得多。

### 1.3 重构自己项目的优先级（如果明天重做）

```
P0 先做：消息链路 + 三层记忆 + delegate_task + 微信网关
P1 再做：权限门控 + 上下文压缩 + fact 提取 + runtime jobs
P2 后做：多 scope 召回 + observation store + corrective recall + triplet
砍掉：早期 academic-style memory 模块拆分（chunking / semantic_index 等可合并）
```

## 2. 评估之难（Agent 评估本身的缺陷）

### 2.1 我们用什么评估

| 评估方法 | 用在哪 | 局限 |
|----------|--------|------|
| **单测 + pytest gate** | 工具函数、压缩、记忆索引等纯逻辑 | 不覆盖 LLM 输出 |
| **Replay fixture**（无 LLM 脚本） | 编排路径验证 | 不能反映真实模型行为 |
| **CC harness gate** | 上下文经济 / read-state / 队列 | 对标 Claude Code，但 Butler 比 CC 多 5+ 能力 |
| **Sim（owner-sim）** | Owner 视角真实使用 | 一次性，每次重写 |
| **手动验收卡** | 委派任务 | Owner 主观判断，不可复现 |
| **LLM-as-Judge** | ⚠ 没用过 | 见 §2.2 |

### 2.2 为什么没用 LLM-as-Judge

**理由**：
1. **循环依赖**：用 LLM 评 LLM，本质是同分布验证，无法检测系统性偏差
2. **维度难定义**：Agent 输出的"好"是多维（准确 / 安全 / 风格 / 成本），LLM 难以加权
3. **实时性差**：每次模型升级，所有历史评判失效

**我们选择的妥协**：**人审 + 业务指标**（Owner 验收通过率、runtime jobs 完成率、Recall@3），不追求"全自动评估"。

### 2.3 评估本身的 3 个隐藏缺陷

| 缺陷 | 现象 | 我们的对策 |
|------|------|------------|
| **指标失真** | Recall@3 高，但用户说"找不到" | 看实际 user feedback（`/反馈` 命令） |
| **过拟合测试集** | 测过的 case 表现好，没测过的崩 | 测试集随真实使用滚动更新（`但ler memory ingest`） |
| **滞后评估** | 模型升级后才知退化 | runtime metrics + 漂移检测（`/诊断`） |

**关键洞察**：**评估的价值不在"测出来多少分"，而在"暴露哪些盲区"**。

## 3. 认知颠覆（对 LLM 能力边界的新认知）

> 这一栏是面试官的"加分项"。讲出真诚的反思比讲"我们做得多好"更有说服力。

### 3.1 关于"规划能力"

**天真认知**：LLM 真的能"规划"——分解任务、调度工具、动态决策。

**实际认知**：**LLM 没有真正的"规划"，是高级的模式匹配**。

具体证据：
- 子代理在长程任务里**经常偏离**最初的 plan（即使 plan 写在 system prompt 里）
- 多步任务里**前 3 步基本准确，第 5 步开始漂移**
- 上下文压力越大，规划越退化为"最近 2 步的局部最优"

**我们的应对**：
- 不让 Agent 做"长程规划"，而是**短程规划 + 短程执行 + 短程验证**
- 关键决策通过 `delegate_task` 拆给子 Loop，每个子 Loop 独立规划（**短上下文 = 强规划能力**）
- 跨子 Loop 的"全局规划"靠 human-in-the-loop（Owner 验收卡）

### 3.2 关于"工具幻觉"

**天真认知**：Function calling 协议会让 LLM 严格按 schema 调用工具。

**实际认知**：**协议约束的是"格式"，不约束"意图"**。

具体证据：
- LLM 会调用 schema 合法的工具但**意图错位**（如用 `read_file` 去"恢复"已删除文件）
- LLM 会**幻觉参数值**（如把日期编造成 "2025-13-45"）
- LLM 会**幻觉工具名**（即使工具不在白名单，也会尝试调用）

**我们的应对**：
- permissions.yaml 是**意图防火墙**（不是格式防火墙）
- read-before-edit 是**事实校验**（防止基于幻觉的 patch）
- 验收卡是**最后一道 human check**

### 3.3 关于"上下文"

**天真认知**：上下文越长越好——所有信息都在窗口里。

**实际认知**：**上下文是负债，不是资产**。

具体证据：
- Token 越长，**单次推理成本越高**（O(N²) attention）
- 上下文越长，**关键信息越被淹没**（lost in the middle 现象）
- 上下文越长，**错误越容易传染**（一个错误信息被反复引用）

**我们的应对**：
- **主动压缩**：token ≥ 阈值 + 消息 ≥ 12 条触发
- **被动压缩**：API 413 触发 `reactive_compact`
- **Fact 提取**：压缩前把决策/约定写回项目 MEMORY（让 SSOT 比 LTM 更可信）
- **锚点重注入**：压缩后从 MEMORY 注入"最近 3 条决策"作为锚点

### 3.4 关于"自建 vs 框架"的真实判断

**天真认知**：自建 = 先进；用框架 = 偷懒。

**实际认知**：**自建 vs 框架不是技术决策，是商业/场景决策**。

| 场景 | 推荐 |
|------|------|
| 多租户 SaaS、需要可视化工作流 | LangGraph / AutoGen |
| 单租户自托管、远程微信原生、极致可控 | 自建 Loop |
| 快速验证 MVP | LangChain |
| 长期维护 3+ 年 | 自建（无 breaking change 风险） |

**关键判断**：**自建的真正价值不是"更先进"，而是"无升级风险 + 全栈可控"**。

### 3.5 关于"Agent 离 AGI 还差多远"

**我的判断**（个人观察，可能不完全对，欢迎反驳）：

- 当前 LLM 在 Agent 场景下，**能力上限 ≈ 高级模式匹配 + 短程规划 + 工具调用**
- 真正的"长程规划 + 创造性问题求解"在我日常使用体感里**还不够稳健**——具体差距多少不好量化
- 多 Agent 协同的"涌现智能"**在我的项目里没观察到**——子代理之间更像"接力"而非真正的"协作"
- **Agent 真正突破需要"工作记忆 + 长期记忆 + 元认知"的工程化**，更大模型肯定有帮助，但单靠 scale 我不乐观

**为什么这个判断重要**：面试官问"你怎么看 Agent 的未来"，回答"还有距离"比回答"马上到来"更显示工程思维——但避免用绝对数字（如"2-3 个量级"），那是没有依据的玄学。
