## 🐛 3 个真实 Bug（从 commit `323862e` 提炼）

1. **`transcript_fts.py` PRIMARY KEY 多余 `)`**：SQLite 早期版本静默吞语法错，导致 FTS 索引重建失败但没日志
   - 教训：第三方容错是双刃剑，必须加显式 health check
2. **`croniter` 缺失被 try/except 吞掉**：job 注册成功但永不触发
   - 教训：静默失败是 Agent 系统隐形杀手，必须"明确失败 or 明确降级"
3. **压缩丢锚点**（自研 bug）：长程任务跑 2h 后忘第 3 轮约定
   - 修法：压缩前抽 fact 写回 MEMORY.md，压缩后锚点重注入

## 🚫 5 个产品边界（不做清单）

- ❌ 不做全量 MCP Host / 市场（只做薄客户端）
- ❌ 不做 IDE 子进程替代 Loop（远程微信不适配）
- ❌ 不做多租户 SaaS（单租户自托管定位）
- ❌ 不用 LangGraph 替换自建 Loop
- ❌ 不做浏览器自动化默认路径（安全边界）

---

## ⚖️ 自建 vs 框架（一句话判断）

> **不是技术决策，是场景决策**。
> 多租户 SaaS / 可视化工作流 → LangGraph；
> 单租户自托管 / 微信原生 / 极致可控 → 自建。
> **自建的真正价值不是"更先进"，而是"无升级风险 + 全栈可控"**。

---

## 🧠 3 个认知颠覆（结尾收口用）

1. **LLM 没有真正的"规划"**——是高级模式匹配；前几步准，多步后漂移
2. **Function calling 约束的是"格式"不约束"意图"**——LLM 会调 schema 合法但意图错位的工具
3. **上下文是负债不是资产**——必须压，但压前必抽 fact 写回 SSOT

## 🗣 开场白（45-60 秒）

> "我最近一年多在做 **WFXM Butler**，一个自托管的多项目 AI 管家。**核心场景是：你在微信上发消息，它能理解意图、跨工具调度、跨项目切换、跨会话记住上下文**。
> 技术上最核心的两个挑战——
> **一是'跨进程工具路由'**：工具数超 100、Agent 横跨 Owner 对话 / dev 子代理 / 外部 MCP，怎么保证既不越权、又不幻觉；
> **二是'多源记忆治理'**：记忆来自三层（Owner 画像 / 项目 MEMORY / 会话 transcript），怎么在压缩、漂移、抽取下不失真。
> 我的解法是'自建 Agent Loop + 文件 SSOT'——**1254 个 Py 文件**，不用 LangGraph，理由是单租户自托管场景下框架自带能力是负债而非资产。
> 今天演示 backlog 已交付 9/10 项，主流程跑完。接下来 2-3 分钟我展开讲第二个挑战——'多源记忆治理'。"

## 🎬 结尾收口（30 秒）

> "总结一下，**WFXM Butler 的核心挑战**就是这两个：
> - **跨进程工具路由**——让 100+ 工具在多 Owner、多子代理、多权限域里安全调度
> - **多源记忆治理**——让三层记忆（Owner / 项目 / 会话）在压缩、漂移、抽取下不失真
> 这两个挑战也是 Agent 行业目前最大的两个工程难题。我的方案是'自建 Loop + 文件 SSOT'。"

## 🔥 高频追问（TOP 5）

| 问题 | 一句话 |
|------|--------|
| **为什么不用 LangGraph？** | 单租户自托管 + 微信原生 + 极致可控的场景下，框架自带能力是负债 |
| **为什么不用 Postgres？** | 文件 SSOT 可 `git diff` 审计；SQLite 索引可 `reindex` 重建 |
| **为什么不用 Claude Code？** | Claude Code 没微信入口 + 多项目切换 + 分层记忆 |
| **Agent 幻觉怎么解决？** | 5 道核心防线（口播）/ 2 层 15 道门（追问展开） |
| **怎么看 Agent 未来？** | 当前 LLM ≈ 高级模式匹配 + 短程规划 + 工具调用；真正突破需要"工作记忆 + 长期记忆 + 元认知"工程化 |
