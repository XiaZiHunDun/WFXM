# 02 · 3-5 分钟深讲脚本

> **用途**：开场白之后，或面试官说"展开讲讲 XX"。按章节切成 4 段，每段 45-90 秒。**口径已修正**：原标"2-3 分钟"是压缩极限，实测节奏 285s ≈ 4-5 分钟；面试官给深讲窗口通常是 3-5 分钟，4 段全讲刚好打满。

---

## 段 1：多源记忆治理（推荐主推，90 秒）

> **"我先讲'多源记忆治理'——这是 Agent 行业目前最大的工程难题之一。**
>
> **挑战**：当 Agent 跨会话、跨项目运行时，记忆会从三个源冒出来——
> - **Owner 层**（`~/.butler/memory/`）：画像、跨项目经验
> - **项目层**（`<workspace>/.butler/memory/MEMORY.md`）：架构、决策、约定
> - **会话层**（`transcript.jsonl`）：对话轨迹
>
> **传统做法**是塞进向量库或数据库。但我们有 4 个观察——
> 1. **数据库做 SSOT 难审计**：Owner 看不到记忆变更，git diff 失效
> 2. **向量检索失真**：Recall@3 高但用户说"找不到"——lost in the middle
> 3. **压缩丢锚点**：第 5 步任务失忆之前的决策
> 4. **漂移不检测**：项目 MEMORY 改了，transcript 没更新，知识分裂
>
> **我的方案**——
> - **SSOT 是文件**（Markdown/JSONL），SQLite 只做**派生索引**（可 `reindex` 重建）
> - **压缩前先抽 fact 写回 MEMORY.md**——让 SSOT 比 LTM 更可信
> - **锚点重注入**——压缩后从 MEMORY 注入"最近 3 条决策"
> - **漂移每日报告**——`todos-pending-drift-daily` job 检测 MEMORY vs todo 的漂移
>
> **成果**：演示 backlog 9/10 已交付，其中 3 项直接对应记忆治理（P0 #1 schema 校验 / P1 #6 去重 / P1 #7 漂移）。"

---

## 段 2：跨进程工具路由（备选主推，90 秒）

> **"再讲'跨进程工具路由'。**
>
> **挑战**：当工具数超过 100，Agent 横跨多个进程边界——
> - Owner 对话进程
> - dev / content / review 子代理进程
> - 外部 MCP 进程（GitHub / Firecrawl / MarkItDown）
>
> **核心问题不是'调用工具'，是'路由工具'**——带权限过滤、上下文裁剪、子代理窄化。
>
> **我口头讲 5 道核心幻觉防线**（好记）——
> 1. `permissions.yaml` allow/deny/ask 三态
> 2. `filter_tools_for_subagent` 子代理工具窄化
> 3. `read-before-edit` patch 前必须 read_file
> 4. `DELEGATE_BLOCKED_TOOLS` 子代理不能再 delegate
> 5. typed schema 严格参数校验
>
> **⚠ 如果面试官追问细节**：真实架构是 **2 层 15 道门**（入站管线 9 步 + 工具执行链 6 层），见 PPT §4 / `02-action.md §4`。能背出完整 15 道说明读过源码——这是加分项。
>
> **3 道异常回退**——
> - `llm_fallback`：主模型 API 失败时切备用 Provider
> - `reactive_compact`：API 413 触发被动压缩
> - 微信 outbox 重试：durable 出站
>
> **最戏剧化的一个 bug**：早期 dev 子代理被委派改某个项目工作区，结果误改项目的 `MEMORY.md`。根因是 `safe_root` 没生效，子代理工具集与父 session 共享。修法是 4 管齐下（safe_root + 子代理窄化 + read-before-edit + 验收卡）。**这类越权尝试修法上线后再未出现过**（参见 `path_safety` 32 个测试文件覆盖）。"

## 段 3：自研 vs 框架（必讲，60 秒）

> **"面试官常问：为什么不用 LangGraph？**
>
> **我的回答**：这不是技术决策，是**场景决策**。
>
> | 场景 | 推荐 |
> |------|------|
> | 多租户 SaaS、可视化工作流 | LangGraph |
> | 单租户自托管、微信原生、极致可控 | **自建** |
> | 快速验证 MVP | LangChain |
> | 长期维护 3+ 年 | 自建（无 breaking change 风险）|
>
> **自建的真正价值不是'更先进'**——
> - **无升级风险**（没有 LangGraph v0.x → v1.x 的 breaking change）
> - **全栈可控**（每个工具调用是 Python 函数，可断点）
> - **白盒审计**（`transcript.jsonl` 全程可读，git diff 能看所有记忆变更）
>
> **代价**：没有 LLM/MCP/记忆的现成生态；需要自己写所有边界 case。但 1254 个 Py 文件都在掌控之内。
>
> **关键判断**：Agent 框架选型不是技术问题，是**商业/场景问题**。"

---

## 段 4：认知颠覆（结尾收口，45 秒）

> **"最后讲讲我对 LLM 能力边界的新认知**——
>
> - **LLM 没有真正的'规划'，是高级模式匹配**——子代理在长程任务里前 3 步准确，第 5 步开始漂移
> - **Function calling 约束的是'格式'，不约束'意图'**——LLM 会调用 schema 合法但意图错位的工具
> - **上下文是负债不是资产**——压缩是必须，但压缩前必须抽 fact 写回 SSOT
>
> **真正的 Agent 突破**需要'工作记忆 + 长期记忆 + 元认知'的工程化，不是更大模型。"
