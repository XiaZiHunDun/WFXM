# 作家部门工作流程

## 部门概述

作家部门负责正文创作，共10名作家（A-J），由主编统筹。

### 架构

```
作家主编（统筹）
├── 作家A：负责阶段1-2（20-40章）
├── 作家B：负责阶段2-3（20-40章）
├── 作家C：负责阶段3-4（20-40章）
├── 作家D：负责阶段4-5（20-40章）
├── 作家E：负责阶段5-6（20-40章）
├── 作家F：负责阶段6-7（20-40章）
├── 作家G：负责阶段7-8（20-40章）
├── 作家H：负责阶段8-9（20-40章）
├── 作家I：负责阶段9-10（20-40章）
└── 作家J：负责阶段10-11（20-40章）
```

### 主编职责

| 职责 | 说明 |
|------|------|
| 任务分配 | 按阶段分配作家任务 |
| 进度监控 | 跟踪每批次完成状态 |
| 格式检查 | 检查提交格式是否符合规范 |
| 跨阶段协调 | 处理阶段间的衔接问题 |
| 向主控汇报 | 定期汇报进度和风险 |

## 创作流程

### 阶段划分

| 阶段 | 章节范围 | 主要任务 |
|------|----------|----------|
| 阶段1 | ch001-ch040 | 世界观建立、主角成长 |
| 阶段2 | ch041-ch080 | 冲突升级、人物登场 |
| ... | ... | ... |

每20章为一个阶段，每阶段结束时提交阶段汇总。

### 创作规范

| 规范项 | 要求 |
|--------|------|
| 每章字数 | 3000±200 字 |
| 章节结构 | 开端→发展→高潮→结尾 |
| 结尾要求 | 留悬念或情绪钩子 |
| 禁止事项 | 大段说教、重复场景、逻辑矛盾 |
| 风格遵循 | 遵守项目全局 style_guide |

### 提交格式

```
03_内容仓库/04_正文/chXXX.md

文件头部元数据：
---
章节: chXXX
版本: v1.0
作者: 作家A
创作时间: YYYY-MM-DD
自评等级: S/A/B
---
```

## 批次管理

### 批次流程

```
作家创作 → 自评S/A/B → 主编格式检查 → 提交审核 → 审核反馈 → 修改 → 复审 → 定稿
```

### 批次规格

| 批次 | 章数 | 产出时间 | 审核周期 |
|------|------|----------|----------|
| 短批 | 5章 | 1天 | 0.5天 |
| 标准批 | 10章 | 2天 | 1天 |
| 长批 | 20章 | 4天 | 2天 |

### 批次追踪

```yaml
批次: B01_ch001-ch010
状态: in_progress
作家: A-J
创作进度: 8/10
审核状态: pending
预计完成: 2026-05-15
```

## 并行执行规则（关键）

### 高效并行模式

**每批次10章，必须10个作家并行（每人一章）：**

| 作家 | 分配章节 | 职责 |
|-----|---------|------|
| 作家A | chN | 独立负责分配的章节 |
| 作家B | chN+1 | 独立负责分配的章节 |
| ... | ... | ... |
| 作家J | chN+9 | 独立负责分配的章节 |

### 禁止模式

- ❌ 一个作家串行修改多个章节
- ❌ 只启动2-3个作家修改10章
- ❌ 作家闲置（每批次必须启动与章数相等的作家数量）

### 启动方式

使用 Agent 工具启动作家，参数：
- `run_in_background: true`（后台并行）
- 携带：章节内容路径、修改任务、问题清单

## 反馈循环

### 接收审核反馈

1. 读取意见仓库中的审核记录
2. 识别问题类型：
   - **P0问题**（逻辑矛盾、设定冲突）→ 立即修复
   - **P1问题**（影响阅读、节奏问题）→ 优先修复
   - **P2问题**（优化建议）→ 纳入批次修复
3. 执行修复
4. 输出修改报告到意见仓库

### 修改完成报告格式

```markdown
## 修改完成报告

### 修改章节
- ch001：修改内容...
- ch002：修改内容...

### 问题修复状态
- 问题1（性别矛盾）：✅ 已修复
- 问题2（衔接断裂）：✅ 已修复
- 问题3（节奏拖沓）：⚠️ 部分优化

### 未解决问题
- 问题4（伏笔不足）：原因...，建议...
```

## 质量评估

### S/A/B评估标准

| 等级 | 完整度 | 处理 |
|------|--------|------|
| S级 | >90% | 直接提交审核，标记高质量 |
| A级 | 70%-90% | 提交审核，附自评说明修改点 |
| B级 | 50%-70% | 修改后再自评 |
| 不合格 | <50% | 打回重写，记录返工 |

### 自评机制

作家完成章节后按S/A/B标准自评，提交审核时附带：
- 自评等级
- 自评意见（指出修改点或亮点）

审核结果与自评对比记入能力档案。

### 能力档案

```yaml
作家A:
  能力评分:
    逻辑一致性: 8
    文笔流畅度: 7
    人物塑造: 7
    伏笔管理: 6
  产出统计:
    完成章节: 40
    返工率: 15%
    通过率: S:20% / A:60% / B:20%
  状态: 正常
```

## 小批快返规则

### 核心原则

- 每10章为一批，完成后立即提交审核
- 发现问题早修正，不累积
- 连续3次返工触发主控介入评估

### 问题早发现清单

| 问题类型 | 发现时机 | 处理方式 |
|----------|----------|----------|
| 逻辑矛盾 | 创作时 | 立即停下，确认后再写 |
| 节奏拖沓 | 自审时 | 精简或合并场景 |
| 人设偏离 | 写完后 | 回溯修正至少3章 |
| 伏笔遗忘 | 阶段末 | 补充或清理伏笔 |

## 对接规范

### 与灵感部门对接

- 读取：基础层.yaml + 深度层.md
- 如有疑问：通过主控协调向灵感部门追问
- 禁止直接向灵感部门发起追问

### 与审核部门对接

- 每10章为一批批次提交
- 主编格式检查后转交审核
- 审核结果反馈：S/A/B + 修改意见

### 与汇总部门对接

- 阶段创作完成后，提交阶段汇总素材
- 配合汇总部门的衔接段修改需求

./run_workflow.sh writer_status

## 紧急备忘

1. **禁止串行**：绝不使用单一作家串行修改多章节
2. **禁止跳过**：绝不在审核后跳过修改主持步骤
3. **高效并行**：每批次必须启动与章数相等的作家数量
4. **小批快返**：每10章一批，发现问题早修正
