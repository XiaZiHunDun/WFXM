# 工作流规则手册

> **目的**：汇总所有工作流规则，规范化各部门协作

## 一、工作流状态机

### 状态定义

| 状态 | 说明 | 可转换到 |
|------|------|---------|
| PENDING | 等待处理 | IN_PROGRESS |
| IN_PROGRESS | 执行中 | COMPLETED / FAILED |
| COMPLETED | 已完成 | VERIFIED |
| VERIFIED | 已验证 | - |
| FAILED | 失败 | PENDING（重试） |

### 核心原则

```
状态文件是唯一真相来源，但必须可验证
真实完成 = 有审核报告文件 AND 问题已修复（如需要）
```

## 二、部门协作规范

### 2.1 灵感部门 → 作家部门

**交接标准**：
- 基础层.yaml：类型/主题/人物/风格禁忌/卖点
- 深度层.md：世界观/时间线/伏笔/叙事结构
- 版本：v1.0 / v2.0 / final

**禁止**：
- 灵感未完成就启动作家
- 版本混乱（同时存在多个版本）

### 2.2 作家部门 → 审核部门

**交接标准**：
- 每10章为一批次
- 自评S/A/B等级
- 格式检查通过

**禁止**：
- 未自评就提交审核
- 跨批次混合提交

### 2.3 审核部门 → 汇总部门

**交接标准**：
- 审核报告已完成
- 问题已修复（如有）
- 无阻塞的P0问题

**禁止**：
- 跳过修改直接标记完成
- 批量"无问题通过"

### 2.4 汇总部门 → 主控

**交接标准**：
- 汇总校验报告已生成
- 版本编号已确定
- 发布记录已创建

## 三、反馈循环规则

### 3.1 强制反馈循环（最高优先级）

```
审核完成 → 汇总意见 → 判定需修改？ → 是 → 启动Agent修复 → TaskOutput验证 → 才能标记完成
                                        → 否 → 标记定稿 → 进入下一批次
```

### 3.2 三条铁律

| 铁律 | 说明 |
|------|------|
| **禁止跳过** | 审核完成后必须进入修改主持流程 |
| **验证闭环** | Agent返回后必须TaskOutput验证，才能改step_status |
| **禁止自改** | 主控不得"自己改文件"，必须通过Agent执行 |

### 3.3 修改主持流程

```
1. 汇总审核意见：按章节合并，去重
2. 判定修改范围：单章节/多章节
3. 启动并行修改：每章分配一个作家
4. 追踪修改状态：记录每个意见的修复状态
5. 重新审核：修改完成后，重新提交原审核员
```

### 3.4 高效并行模式（强制）

```
10章审核 → 汇总意见 → 分配给10个作家（每人一章）→ 10作家并行修改 → 重新审核 → 定稿
```

**禁止模式**：
- ❌ 作家A串行修改ch001-ch010（10章）
- ❌ 只启动作家A、B两人修改10章

## 四、迭代终止条件

### 4.1 大纲审核

| 条件 | 说明 |
|------|------|
| 全量审核员一致通过 | 可终止 |
| 3轮上限 | 超限→人工仲裁 |

### 4.2 正文审核

| 条件 | 说明 |
|------|------|
| 未解决意见≤5条且无逻辑硬伤 | 可终止 |
| 2轮上限 | 超限→人工仲裁 |

### 4.3 死锁处理

```
条件：超迭代上限
处理：
1. 记录当前状态到SESSION_HANDOFF
2. 标记需要人工仲裁
3. 通知主控（用户）
```

## 五、验证规则

### 5.1 审核报告存在性检查

```bash
# 检查批次的审核报告是否存在
for batch in ["ch001-ch010", "ch011-ch020", ...]:
    report_file = f"06_意见仓库/04_正文_审核/{batch}_审核.md"
    if not exists(report_file):
        raise Error(f"批次{batch}标记为完成但无审核报告")
```

### 5.2 问题修复验证

对于状态为"需修改"的批次：

```bash
# 读取源文件，验证问题已修复
content = read(source_file)
if issue.type == "重复内容":
    # 验证重复段落已删除/精简
elif issue.type == "逻辑矛盾":
    # 验证矛盾已解决
```

### 5.3 进度异常检测

| 异常 | 阈值 | 处理 |
|------|------|------|
| 审核速度过快 | >5批次/分钟 | 警报：可能未认真审核 |
| 批量"无问题" | 连续3批100%通过 | 警报：建议抽检 |
| 修复速度过快 | >10章/分钟 | 警报：可能未认真修复 |

## 六、问题优先级定义

| 优先级 | 定义 | 处理要求 | 阻塞定稿 |
|--------|------|---------|---------|
| P0 | 逻辑矛盾、重大重复、关键角色OOC | 必须修复 | ✅ 阻塞 |
| P1 | 叙事断裂、细节不一致、非关键重复 | 应修复 | ⚠️ 影响 |
| P2 | 轻微优化建议、文笔润色 | 建议修复 | ❌ 不阻塞 |

## 七、质量评估标准

### 7.1 审核质量等级

| 等级 | 完整度 | 处理 |
|------|--------|------|
| S级 | >90% | 直接进入下一环节 |
| A级 | 70%-90% | 主编小幅调整 |
| B级 | 50%-70% | 返回重做（不计入迭代） |
| 不合格 | <50% | 打回重做（计1次迭代） |

### 7.2 读者质量等级

| 等级 | 完整度 | 处理 |
|------|--------|------|
| S级 | >90% | 亮点突出，毒点少 |
| A级 | 70%-90% | 有亮点，有毒点但可接受 |
| B级 | 50%-70% | 毒点明显，需修改 |
| 不合格 | <50% | 弃书指数≥8，直接打回 |

### 7.3 弃书率阈值

| 弃书指数 | 处理 |
|---------|------|
| ≤5 | 合格，进入下一批 |
| 6-7 | 警告，需修改 |
| ≥8 | 不合格，打回重新创作 |

## 八、会话交接规则

### 8.1 交接时机

每次会话结束时，必须执行：
- 本批次审核报告已生成
- 报告文件已验证存在
- 问题修复已抽样验证
- workflow_state.json已更新
- 无阻塞的subagent任务
- 交接文件已生成

### 8.2 交接文件格式

```markdown
# 会话交接：{日期时间}

## 完成状态
- 当前批次：{chXXX-chYYY}
- 状态：{step_status}
- 发现问题：{N}条（P0:{N}/P1:{N}/P2:{N}）

## 待处理
- 待修复问题：{N}条
- 待审核批次：{N}个

## 注意事项
- {任何需要接续者注意的事项}
```

## 九、版本迭代规则

### 9.1 灵感版本迭代

```
v1.0 → v2.0 → v3.0 → final
最多3轮迭代，超限主控仲裁
```

### 9.2 汇总版本迭代

| 版本 | 触发条件 | 内容要求 |
|------|---------|---------|
| v1.0 | 初次汇总完成 | 基础内容整合，结构完整 |
| v2.0 | 审核发现需修改 | 修复逻辑矛盾，补充过渡 |
| v3.0 | 二次审核通过 | 细节打磨，风格统一 |
| final | 三审通过+校验完成 | 可发布版本 |

### 9.3 发布版本命名

```
{项目名}_{版本标识}_{日期}
示例：全文终稿_星陨纪元_v1.0_20260514
```
