# 正文审核验收标准

> **目的**：确保每批次审核可验证、可追溯，防止"虚假进度"

---

## 一、审核完成定义

### 有效审核条件（必须全部满足）

```
□ 审核报告文件存在于 `06_意见仓库/04_正文_审核/`
□ 审核报告命名格式：`ch{START}-{END}_审核.md`
□ 审核报告包含：章节列表、问题类型、优先级、状态
□ 对"需修改"的批次：
  □ 问题已修复（读取源文件验证）
  □ 修复记录已更新至 workflow_state.json
```

### 禁止行为

```
❌ 禁止：快速浏览章节后标记为"已审核"
❌ 禁止：仅更新state文件但无审核报告
❌ 禁止：批量标记"无问题通过"但无抽检
```

---

## 二、批次审核流程

### 标准流程

```
1. 启动审核Agent
   └── 指定章节范围：ch{START}-{END}

2. Agent审核并生成报告
   └── 输出至：06_意见仓库/04_正文_审核/ch{START}-{END}_审核.md

3. 验收检查
   └── 主控验证报告文件存在且内容完整

4. 问题修复（如需要）
   └── 作家修复 → 修复验证 → 更新issues_found

5. 批次标记完成
   └── 更新workflow_state.json的completed列表
```

## 三、问题优先级定义

| 优先级 | 定义 | 处理要求 | 阻塞定稿 |
|--------|------|---------|---------|
| P0 | 逻辑矛盾、重大重复、关键角色OOC | 必须修复 | ✅ 阻塞 |
| P1 | 叙事断裂、细节不一致、非关键重复 | 应修复 | ⚠️ 影响 |
| P2 | 轻微优化建议、文笔润色 | 建议修复 | ❌ 不阻塞 |

## 四、进度追踪规则

### 真实进度 = 审核报告数量

```bash
# 验证命令
ls 06_意见仓库/04_正文_审核/*_审核.md | wc -l

# 应等于：实际完成审核的批次数量
```

### State文件一致性检查

```python
# 伪代码：一致性验证
state_completed = read_json("workflow_state.json")["review_queue"]["completed"]
audit_files = list_files("06_意见仓库/04_正文_审核/*_审核.md")

for batch in state_completed:
    if batch not in audit_files:
        raise "状态文件与审核记录不一致！"
```

## 五、交叉验证机制

### 修复验证（抽样比例≥10%）

对于"需修改"的批次，随机抽取10%的问题章节，读取源文件验证：

```
验证点：
□ 问题描述位置正确
□ 修复内容已应用
□ 修复后无引入新问题
```

### 进度异常检测

| 异常 | 阈值 | 处理 |
|------|------|------|
| 审核速度过快 | >5批次/分钟 | 警报：可能未认真审核 |
| 批量"无问题" | 连续3批100%通过 | 警报：建议抽检 |
| 修复速度过快 | >10章/分钟 | 警报：可能未认真修复 |

## 六、角色职责分离

| 角色 | 职责 | 禁止 |
|------|------|------|
| 审核员 | 找问题，生成报告 | 自己修复自己审核 |
| 修复者 | 根据报告修复问题 | 修改审核结论 |
| 验收员 | 验证修复、确认完成 | 跳过验证步骤 |
