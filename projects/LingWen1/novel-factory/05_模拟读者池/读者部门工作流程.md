# 读者部门工作流程

## 概述

读者部门由20位模拟读者组成，负责对小说正文进行真实阅读体验反馈，作为审核部门的补充视角。

## 并行阅读规则

### 批量任务分配

```
主控调度 → 批量分配 → 读者群并行 → 评论汇总 → 意见仓库
```

**分配策略：**
| 任务规模 | 并行人数 | 分配方式 |
|---------|---------|---------|
| 1-5章 | 1-3人 | 按读者专长分配 |
| 6-10章 | 5-10人 | 每1-2章一人 |
| 11-20章 | 10-15人 | 每1-2章一人，跨读者协作 |
| 21+章 | 15-20人 | 全员并行，分段覆盖 |

### 并行执行模式

```
任务：阅读ch001-ch010
执行方案：
  读者A → ch001-ch002
  读者B → ch003-ch004
  读者C → ch005-ch006
  读者D → ch007-ch008
  读者E → ch009-ch010
  ...（按读者专长扩展）
```

### Subagent启动方式

使用 Agent 工具启动读者，参数：
- `run_in_background: true`（后台并行）
- 携带章节内容和输出路径

### 标准模板

```markdown
## 读者评论：{章节号}

### 读者信息
- 读者：{读者编号}
- 阅读时间：{完成时间}
- 阅读场景：{场景}

### 总体评价
- 评分：{X}/10
- 亮点：{简述}
- 毒点：{简述}

### 详细评论
（按阅读体验逐段评论，包含）
- 情节评价
- 人物评价
- 文笔评价
- 节奏评价

### 情绪曲线
- 期待感：▓▓▓▓▓░░░░░（1-10）
- 代入感：▓▓▓▓▓▓▓░░░（1-10）
- 惊喜度：▓▓▓▓░░░░░░（1-10）

### 弃书指数
{1-10}（超过7请说明原因）

### 核心问题（严重时填写）
- P0问题：{逻辑硬伤/降智描写}（如有）
- P1问题：{人设崩塌/情节漏洞}（如有）
- P2建议：{优化方向}（可选）
```

### 评分维度说明

| 维度 | 说明 |
|------|------|
| 期待感 | 看完一章后想继续读的冲动 |
| 代入感 | 与主角的共鸣程度 |
| 惊喜度 | 超出预期的反转/设定 |

## 紧急反馈机制

### P0问题定义

发现以下问题时立即上报主控：
1. **逻辑硬伤**：前后矛盾、事实错误
2. **降智描写**：主角/配角行为不符合智商
3. **价值观偏差**：涉及政治/伦理红线
4. **事实错误**：专业领域明显错误

### 紧急上报格式

```markdown
## [紧急] P0问题上报

**章节：** chXXX
**问题类型：** {硬伤/降智/违规/错误}
**问题描述：** {具体描述}
**影响范围：** {范围}
**建议处理：** {建议}

**上报时间：** {时间戳}
**紧急程度：** 最高优先级
```

### 上报流程

```
发现P0问题 → 立即上报主控 → 主控调度审核部门复核 → 确认后启动修复流程
```

## 与审核部门的协作

### 双重审核机制

```
作家提交 → 审核部门（技术审核）→ 读者部门（市场审核）→ 汇总部门
```

### 读者反馈价值

- 真实阅读体验（非技术角度）
- 市场受众视角
- 情感共鸣检测
- 弃书点识别

---

## 质量保证

### 读者评论质量标准

| 等级 | 要求 |
|------|------|
| 优秀 | 覆盖所有评分维度，有具体案例支撑 |
| 合格 | 覆盖主要维度，评论清晰 |
| 待改进 | 评论过于简略或缺乏实质性内容 |

### 评论审核机制

- 每篇评论由主编抽查
- 连续3篇不合格，调整读者任务分配
- 严重失职（如敷衍评论）替换读者

---

## 记忆与档案

读者部门维护每位读者的：
- 阅读履历（完成章节）
- 评论质量记录
- 专长领域
- 当前状态（空闲/阅读中/已完成）
