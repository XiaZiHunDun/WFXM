# 小说生产公司 · 完整设计方案

## 一、整体架构与职责定义

项目模拟一家工业化小说生产公司，下设 **七个部门**，每个部门由多个具有独立画像和记忆的 Subagent 扮演，借助 Claude Code 实现多智能体协作。

| 部门 | 职责 | 输出物 |
|------|------|--------|
| 灵感部门 | 提供类型、主题、世界观、基调、体量、禁忌等创作方向，输出结构化立项文件 | 灵感库/结构化文件 |
| 作家部门 | 创作逐层大纲和正文，每位作家有独立画像+记忆，负责不同卷/阶段 | 内容仓库四层文档 |
| 内容仓库 | 分层存放全文总体大纲、卷大纲、阶段大纲、章节正文 | 结构化内容文件 |
| 审核部门 | 分层审核（大纲+正文），每位审核员独立画像，侧重逻辑/设定/人设/合规等 | 审核意见 |
| 读者部门 | 模拟真实读者阅读正文，产生评论和意见 | 读者评论意见 |
| 意见仓库 | 统一归档所有审核意见、读者评论、修改记录 | 意见文档（可回溯） |
| 汇总部门 | 对定稿章节进行阶段汇总→卷汇总→全文汇总，产出最终发布稿 | 汇总仓库各层汇总稿 |

---

## 二、完整工作流（25步闭环）

### 阶段一：立项
1. **灵感部门** 生成结构化灵感文件，提交至灵感库。  
2. **作家部门**（主作家）根据灵感文件生成《全文总体大纲》，存入内容仓库。

### 阶段二：全文大纲迭代
3. **审核部门** 对全文总体大纲进行审核，输出审核意见至意见仓库。  
4. **作家部门** 根据审核意见修改全文总体大纲。  
5. 重复 3-4，直到 **全量审核员一致通过**（无重大意见且评分达标）。

### 阶段三：卷大纲迭代
6. **作家部门** 根据全文总体大纲，分配作家生成各**卷大纲**（可根据画像匹配）。  
7. **审核部门** 对各卷大纲进行审核，输出意见。  
8. **作家部门** 根据意见修改。  
9. 重复 7-8，直到所有卷大纲通过审核。

### 阶段四：阶段大纲迭代
10. **作家部门** 根据卷大纲拆分**阶段大纲**。  
11. **审核部门** 审核阶段大纲。  
12. **作家部门** 修改。  
13. 重复 11-12，直至通过。

### 阶段五：正文创作与双轨反馈
14. **作家部门** 根据阶段大纲逐章撰写正文，每完成一章提交内容仓库。  
15. **读者部门** 阅读新完成章节，提交读者意见。  
16. **审核部门** 同步对正文进行技术性审核（伏笔、一致性、设定冲突、合规等），提交审核意见。  
17. **作家部门** 根据读者意见与审核意见，修改章节。  
18. 重复 15-17，直到 **该章节未解决意见总数 ≤ 5 条**，且 **审核员确认无逻辑硬伤**，标记为“定稿”。当一阶段内所有章节定稿后，该阶段定稿。

### 阶段六：分层汇总与终审
19. **汇总部门** 对已定稿的阶段内所有章节进行拼接、过渡润色，生成《阶段汇总》，存入汇总仓库。  
20. **审核部门** 审核阶段汇总，确保连贯性和风格统一，输出意见。  
21. **作家部门** 根据意见微调（必要时回修正文），直至通过，阶段汇总定稿。  
22. 当一个卷内所有阶段汇总完成，**汇总部门** 生成《卷汇总》，审核循环同阶段汇总。  
23. 所有卷汇总定稿后，**汇总部门** 生成《全文汇总》（最终成品）。  
24. **审核部门** 终审（全文一致性、前后照应、市场定位），通过后标记版本号，移至汇总仓库发布区。

### 阶段七：归档闭坏
25. **意见仓库** 统一归档所有灵感、审核意见、读者评论、修改日志。**作家部门** 更新个人记忆文档，记录本次协作习得的设定、人物细节、常见问题。可开启下一项目或续作。

> **可选增强步骤**（Kimi建议）：  
> 步骤0：在全文总体大纲后，先写“样本章节”（第1章），经审核+读者确定风格基线，再进入批量生产。  
> 步骤X：汇总部门在合成卷汇总时，运行一致性检测（人物年龄、时间线、地名），输出 `consistency_report.md`。

- 直白，给方案，不模棱两可

## 五、核心运行规则

### 1. 迭代终止条件
- **大纲类**：全量审核员一致通过（无重大不通过项）方可进入下一层。
- **正文类**：单章未解决意见 ≤ 5 条，且审核员确认无逻辑硬伤、读者无“弃书级”吐槽。
- **死锁防护**：大纲每层迭代最多 3 轮，正文最多 2 轮读者反馈循环；超限由“总编Agent”强制仲裁，人工决策。

### 2. 版本管理
每次修改生成版本记录：`原内容 → 修改意见 → 修改后内容`，作为独立文件或追加到意见仓库对应目录，文件名含时间戳或版本号，保证全程可回溯。

### 3. Subagent 调度与权限
- **创作权**：大纲和正文只有作家 Subagent 可编辑。
- **审阅权**：审核员、读者只能读取并输出意见，不可直接改动内容文件。
- **并行规则**：无依赖的同层级任务（不同卷大纲、同卷下不同章节）可并行分配多个作家；审核和读者可批量并行输出意见，统一收集后提交给作家。
- **协作契约**：所有作家必须遵守项目全局 `style_guide`（已内嵌于灵感输出和作家 Prompt），确保风格统一。

### 4. 调度总控（可选）
可额外设计一个**总编Subagent**或**调度脚本**，按照工作流步骤依次调用各部门 Subagent，读取状态文件，判断终止条件，自动推进或报警。

## 六、Claude Code 落地实践指南

### 1. Subagent 启动方式
**方案 A：多终端并行（推荐生产环境）**
```bash
# 终编1：启动作家A创作卷1大纲
claude --prompt "你是作家墨痕，依据人设和全文总体大纲撰写卷1大纲..." \
  --file 02_作家工作室/作家A/作家人设.md \
  --file 03_内容仓库/01_全文总体大纲/全文总体大纲.md

# 同时终编2：启动作家B创作卷2大纲
```
利用多个终端会话实现并行创作。

**方案 B：单会话角色切换（适合串行审核）**
在同一个会话中，通过清晰指令切换身份：
```
现在进入【审核员模式】。请读取 04_审核员工作室/审核员1/审核员人设.md。
审核对象：03_内容仓库/02_卷大纲/卷1/卷大纲.md。
输出意见到 06_意见仓库/审核意见/卷大纲/卷1_审核意见_v1.md。
```

**方案 C：编排脚本（最自动化）**
编写 `run_workflow.sh`，利用 `expect` 或简单 `echo` 向 Claude Code 标准输入发送指令序列，自动推进流程。

### 2. 上下文窗口管理（Kimi 警告）
长篇小说全文汇总会超出上下文限制，必须采用**逐层聚合**：
- 汇总阶段：只读本阶段所有章节目录，逐章浓缩摘要后合并。
- 汇总卷：读取阶段汇总而非原始章节。
- 汇总全文：读取卷汇总和前言/后记，不直接加载全部正文。

可在汇总指令中明确：“请仅读取当前层级所需的子节点文件，不要展开全文”。

### 3. MVP 起步策略
首次跑通建议：
- 作家：1名，审核员：1名，读者：2名。
- 先跑单卷，不启用并行。
- 每层审核只做 1 轮，不通过则人工介入。
- 用单终端角色切换模式体验完整链路。
跑通后再扩展作家池、启用并行与自动迭代。

## 七、可选增强（可后续引入）

1. **质量评分体系**：每个大纲、正文输出后自动打分，低于阈值强制重写或触发人工复核。  
2. **风格一致性校验 Agent**：独立检查跨章节文笔、人物语气、世界观细节的一致性，输出警告。  
3. **冲突检测报告**：汇总阶段自动生成 `consistency_report.md`，列出人物年龄矛盾、时间线错误等。  
4. **读者多样性调度**：为不同章节随机分配不同类型的读者组合，模拟不同市场反馈。  
5. **全文发布前读者测评**：终稿生成后，邀请全体模拟读者进行一次“全书阅读体验反馈”，反馈存入灵感库供下个项目使用。
