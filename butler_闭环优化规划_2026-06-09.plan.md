# Butler v4 闭环优化规划

> **类比**：模型训练 = 前向传播(代码实现) + 损失计算(LangFuse/测试) + 反向传播(根据评估结果优化)
>
> 生成日期：2026-06-09

---

## 〇、分析总览

### 系统现状一句话

Butler v4 的**结构性保证**（权限、边界、终止、隔离）已全面实现且测试绿灯；
但**质量闭环**（评估→反馈→自动优化）仍是**单向写入**——数据只流入 LangFuse，不回流。
类比模型训练：**前向传播的计算图已搭好，损失函数已定义，但梯度还没有回传**。

```
┌─────────────────────────────────────────────────────────┐
│                    Butler v4 训练环路                      │
│                                                           │
│   [种子数据]──→[前向传播(代码实现)]──→[损失计算]──→ ??? │
│    T01-T10        agent_loop          LangFuse            │
│    Experiences     dev_engine          pytest              │
│    Corpus          memory              eval_scoring        │
│                    gateway                                 │
│                                                           │
│   ✅ 前向传播：结构完整，8/10 定理有真实 checker          │
│   ✅ 损失函数：eval_bridge + scoring + MB/B benchmarks    │
│   ❌ 反向传播：无 eval→behavior 反馈机制                  │
│   ❌ 种子数据：经验库为空，T02/T07 为 stub               │
│   ❌ 部署自动化：LangFuse 未实际运行，升级需手动          │
└─────────────────────────────────────────────────────────┘
```

## 三、反向传播策略（评估→优化闭环）

### 3.1 当前状态

```
                    ┌──── 单向 ────┐
 Runtime ──trace──→ LangFuse       │  ❌ 无回读
 pytest  ──push──→  LangFuse       │  ❌ 无回读
 dev_delegate ──→ experiences.json  │  ✅ 唯一闭环
                    └──────────────┘
```

**唯一的真正反馈环**：成功的 dev delegate → `extract_experience_candidate()` → `coding_experiences.json`。
其余所有评估数据（LangFuse traces、scores、pytest 结果）都是**可观测但不可行动**。

### 3.2 闭环策略设计

#### 策略 A：评分阈值触发器（自动反馈）

```
LangFuse scores ──(read back)──→ 阈值判断 ──→ 自动动作
                                      │
                    ┌─────────────────┼────────────────────┐
                    ▼                 ▼                    ▼
              调整 prompt        更新 half-life      标记经验为低效
              (template var)     (decay 参数)        (experience TTL)
```

- 当 `intent_accuracy < 0.6` 持续 N 轮 → 触发 prompt 审查建议
- 当 `memory_effectiveness.H_1 < 0.4` → 建议调高 `BUTLER_MEMORY_RECALL_K`
- 当 `E_d > 0.2` → 建议放宽 `DECAY_KILL_THRESHOLD`

#### 策略 B：经验生命周期管理（数据质量）

```
experiences.json ──(eval)──→ 有效性评分 ──→ 续期/降权/淘汰
```

- 经验被 `process_task` 选中 → 任务成功 → 续期 TTL
- 经验被选中但任务失败 → 降低置信度
- 置信度低于阈值 → 标记 deprecated

#### 策略 C：CI 自动推送（消除手动断层）

```
CI pytest ──(on schedule)──→ corpus_run_to_scores ──→ LangFuse
                             memory_eval ──→ LangFuse
                             dev_benchmark_to_scores ──→ LangFuse
```

## 四、LangFuse 共享基础设施方案

### 4.1 需求分析

Butler 管家系统架构：

```
LangFuse (共享基础设施)
├── butler-v4 (管家自身)
├── 灵文1号 (管家管理的项目)
├── 其他项目 A (可能是 agent)
└── 其他项目 B (可能是 agent)
```

### 4.2 推荐方案：单实例 + 多 Project + 动态 Key

| 层级 | 方案 |
|------|------|
| **LangFuse 实例** | 单个 Docker Stack（当前 `deploy/langfuse/`） |
| **Organization** | `butler-org`（已有） |
| **Project 划分** | 每个 Butler 管理的项目 → 一个 LangFuse Project |
| **API Key** | 每个 Project 独立 key，由 Butler 项目配置管理 |
| **Trace 标签** | `project_name` + `tenant` 注入 trace metadata |

### 4.3 实现步骤

1. **LangFuse 多项目初始化 API** — 扩展 `langfuse-setup.sh` 支持创建多个 project
2. **项目级 LangFuse 配置** — `butler/config/` 支持 per-project `langfuse_public_key` / `langfuse_secret_key`
3. **Tracer 动态切换** — `langfuse_tracer.py` 根据当前 project 选择对应 client
4. **Metadata 注入** — 每个 trace 自动携带 `project_name`, `tenant`, `session_source`
5. **共享 Dashboard** — 文档指导如何在 LangFuse UI 创建跨项目视图

## 五、一键部署与自动更新方案

### 5.1 现状问题

| 问题 | 描述 |
|------|------|
| LangFuse 未启动 | `deploy-new-env.sh --langfuse` 未被执行过 |
| 代码更新后需手动部署 | `butler-gateway-ops.sh upgrade` 是手动命令 |
| `[observability]` extra 不在 `[all]` 中 | `pip install -e ".[all]"` 不含 langfuse |
| LangFuse 镜像未锁版本 | 用 `:latest` 有破坏性升级风险 |
| 无自动化部署触发 | CI 只跑测试，不触发部署 |

### 5.2 方案设计

```bash
# 统一入口脚本 butler-deploy.sh
butler-deploy.sh                   # 检测环境类型自动选择
  ├── --init          # 全新部署 = deploy-new-env.sh + langfuse + systemd
  ├── --update        # 增量更新 = git pull + pip + restart + langfuse upgrade + verify
  ├── --langfuse-only # 只管理 LangFuse 栈
  ├── --status        # 全栈状态（Butler + LangFuse + systemd）
  └── --rollback      # 回滚到上一版本（git + pip + restart）
```
